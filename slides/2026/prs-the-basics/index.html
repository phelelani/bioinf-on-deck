  • Choice of data resource and study design:
    • Sample size
    • Experimental question
    • Availability of pre-existing data
    • Ease of new data can be collection
  • Choice of genotyping platform:
    • Depends on many factors
    • Guided by the purpose of the GWAS
  • Next-generation sequencing methods
    • Whole Genome Sequencing (WGS)
    • Whole Exome Sequencing (WES)
  • Microarrays for common variants
    • Most commonly used method
    • Lower cost
  • GWAS input files:
    • Anonymized individual IDs
    • Coded family
    • Sex
    • Phenotype
    • Covariates
    • Genotype calls
    • Info on genotyping batch
  • Quality control:
    • Remove rare/monomorphic variants
    • Remove variants not in Hardy–Weinberg equilibrium
    • Filter missing SNPs
    • Identify and remove genotyping errors
    • Match phenotypes with genetic data
    • Population strata and PCA
  • Variants usually undergo phasing and are imputed:
    • Sequenced haplotype reference panel: 1000 Genomes Project
    • Statistical inference of genotypes not been assayed directly
    • Increase statistical genomic coverage
    • Facilitates meta-analysis of studies
    • Use of genotyping obtained with different panels
    • Increasing genetic markers that can be tested for association
  • Genetic association tests for each genetic variant
  • Use an appropriate model:
    • Additive
    • Non-additive
    • Linear
    • Logistic regression
  • Depends on phenotype:
    • Continuous or binary
  • Covariates included
  • Confounders corrected for
  • Multiple testing needs to be controlled
  • Output inspected for unusual patterns
  • Multiple smaller cohorts are combined
  • Summary statistics are generated

5. Association Testing

5. Association Testing


  • Regression analysis performed for every SNP - each individual as a data point
  • SNP ID #1 association with BMI
    • Two alleles: A & T
    • Possible combinations: AA AT TT
  • Plot number of T alleles vs BMI
  • Fit a line through the data
  • Horizontal line: NO association
  • Slope: association present
  • Record P-Value
    • Measure of the likelihood that the association found in the distribution of data points was due to chance given that the given that there is no association between the SNP and BMI
  • Record the slope of the regression line
    • Effect size or impact
  • Determine statistical significance of the association
    • Is the association significant?
    • Results with P-Value less than 0.05 (5%) considered significant
    • Can also have false positives - random chance
Correcting for the increased error rates in hypothesis

5. Association Testing

5. Association Testing

  • When conducting a GWAS, genotypes of variants that are physically close together are not independent as they tend to be in linkage disequilibrium (LD)
    • LD: Nonrandom association between two or more loci in a population
    • Strength of correlation between two variants is a function of the distance between them - the closer two variants are, the stronger the LD.

Polygenic [Risk] Scores

“An estimate of an individual’s genetic liability to a trait or disease, calculated according to their genotype profile and relevant genome-wide association study (GWAS) data”

~ Choi et al. (2020)


\[ \mathrm{PRS}_i = \sum_{j=1}^{M} \beta_j \cdot G_{ij} \]
  • \(\mathrm{PRS}_i\): Polygenic risk score for individual \(i\)
  • \(M\): Total number of SNPs used in the score
  • \(\beta_j\): Effect size of SNP \(j\) from a GWAS (log-odds ratio or regression coefficient)
  • \(G_{ij}\): Genotype dosage for individual \(i\) at SNP \(j\) (0, 1, or 2 risk alleles)

  • Sum of risk alleles corresponding to a phenotype of interest in each individual
  • Weighted by the effect size estimate of the most powerful GWAS on the phenotype

Docker & Singularity/Apptainer

  • Popular platform for building & sharing containers
  • Linux-based software containers
  • Containers built from a "Dockerfile"
  • Application (+ dependencies) packaged into a unit
  • Major drawbacks:
    • Security risk on shared HPC environments
    • Requires root privileges to run

  • Designed specifically for scientific and HPC use
  • Built on the concept of "Mobility of Compute"
  • Key Features:
    • Secure by design
    • No root privileges required
    • Runs with user's existing privileges
    • Ideal for shared HPC environments

Scaling, Documentation & Sharing

Reproducible Science Depends on Workflows that:

1. Scale consistently
2. Fully version-controlled
3. Shared with code & container

Workflow Scaling:

Ensures consistent results across different compute environments

  • Scaling lets a workflow run identically on a laptop, HPC cluster, or cloud
  • Containerisation (Docker/Singularity) and workflow engines:
    • Ensure the same code & environment regardless of where or how big the system is
  • Parallel execution and scheduler portability prevent human errors may arise
  • Results are stable and repeatable across different hardware, scales, and institutions

Version Control:

Captures exact states of code, parameters, and environments

  • Git records every change, enabling precise tracking:
    • Scripts, configs, reference versions, and parameters
  • Workflow engines often track versions of modules, containers, and dependencies
  • You can reproduce exactly what happened by checking out a tagged version of the repository
  • You can re-run any past version of a workflow and obtain consistent outputs

Sharing:

Ensures others can run the exact same workflow

  • Sharing workflows through GitHub, nf-core, Dockstore, etc.
  • Allows others to access your code and the environment specifications
  • Standard workflow languages (Nextflow, WDL, Snakemake, CWL) make pipelines portable across systems
  • Shared containers and config profiles ensure that collaborators run the same versions of tools
  • Other researchers can replicate or extend your analysis with confidence
